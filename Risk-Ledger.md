## 二、熔断机制（S01-QA-038）

> **状态**: ✅ 已完成

### 2.1 熔断机制概述

**机制目的**:
当项目遭遇重大风险时，及时"止血"，避免问题扩大化。熔断机制定义了明确的触发条件和强制执行的应对动作，确保在危机时刻有章可循。

**三级预警体系**:

| 预警等级 | 颜色 | 严重程度 | 核心动作 |
|---------|------|---------|---------|
| 一级 | 🟡 黄色 | 警告 | 召开专家会诊，加强监控 |
| 二级 | 🔴 红色 | 危险 | 暂停更新，紧急修复 |
| 三级 | ⚫ 黑色 | 致命 | 全面停更，重新审查全文 |

---

### 2.2 黄色预警（一级熔断）

#### 2.2.1 触发条件

当满足以下任一条件时，触发黄色预警：

| 序号 | 触发条件 | 数据来源 | 阈值 |
|------|---------|---------|------|
| Y01 | 订阅连续下跌 | 平台后台数据 | 连续3章订阅环比下跌 > 10% |
| Y02 | 读者差评激增 | 评论区监测 | 24小时内负面评论 > 总评论30% |
| Y03 | 设定矛盾被指出 | 读者反馈 | 同一设定矛盾被3人以上指出 |
| Y04 | 存稿低于警戒线 | 内部监控 | 存稿 < 5章 |
| Y05 | 爽点密度不足 | 内容审计 | 连续5章无明显爽点 |
| Y06 | 质量指标下滑 | 质量报告 | 错别字率连续2周 > 3个/万字 |
| Y07 | 角色OOC投诉 | 读者反馈 | 同一角色OOC被5人以上指出 |

#### 2.2.2 应对动作

**触发黄色预警后，必须执行以下动作**：

```
┌──────────────────────────────────────────────────────────────┐
│                     黄色预警应对流程                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1: 确认预警（30分钟内）                                │
│  ├── 核实数据，排除统计误差                                   │
│  ├── 确认是否为真实风险                                       │
│  └── 记录预警日志                                            │
│                                                              │
│  Step 2: 召开专家会诊（2小时内）                             │
│  ├── 召集相关专家（PM + 主责专家 + QA）                       │
│  ├── 分析问题根因                                            │
│  ├── 评估影响范围                                            │
│  └── 制定应对方案                                            │
│                                                              │
│  Step 3: 执行应对（24小时内）                                │
│  ├── 按方案执行调整                                          │
│  ├── 加强相关监控                                            │
│  └── 准备升级预案（若情况恶化）                               │
│                                                              │
│  Step 4: 效果评估（72小时后）                                │
│  ├── 评估应对效果                                            │
│  ├── 若好转：解除预警，记录经验                               │
│  └── 若恶化：升级为红色预警                                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键原则**：
- 黄色预警期间**不暂停更新**，但需加强监控
- 所有相关章节需经过额外审核
- 预警状态每24小时评估一次

```
│  ├── 【强制】全面停止更新                                     │

│  ├── 发布"重大维护公告"                                       │
│  └── 保护现有素材和数据                                       │
│                                                              │
│  Step 2: 全面审查（1-3天）                                   │
│  ├── 对全文进行逐章审查                                       │
│  ├── 如涉及平台红线：识别所有问题内容                          │
│  ├── 如涉及法律风险：收集相关证据                              │
│  ├── 如涉及设定崩溃：梳理所有矛盾点                            │
│  └── 形成完整的问题清单                                       │
│                                                              │
│  Step 3: 危机评估（与用户沟通）                              │
│  ├── 向用户汇报完整情况                                       │
│  ├── 评估项目是否可挽救                                       │
│  ├── 讨论可能的解决方案：                                     │
│  │   ├── 方案A：大规模修改后继续                              │
│  │   ├── 方案B：从某节点重启                                  │
│  │   └── 方案C：终止项目                                      │
│  └── 由用户做出最终决策                                       │
│                                                              │
│  Step 4: 执行决策                                            │
│  ├── 若继续：制定详细修复计划，预计时长1-4周                   │
│  ├── 若重启：确定重启节点，保留可用素材                        │
│  └── 若终止：妥善收尾，保存经验教训                            │
│                                                              │
│  Step 5: 长期复盘                                            │
│  ├── 深度分析问题根因                                         │
│  ├── 更新所有风险管理文档                                     │
│  ├── 建立更严格的预防机制                                     │
│  └── 记录完整的经验教训                                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键原则**：
- 黑色预警是**最高级别熔断**，意味着项目存续受到威胁
- 所有决策必须由用户亲自确认
- 不得在问题未解决前恢复更新
- 即使最终终止项目，也要完成复盘

**黑色预警公告**（必须发布）：
> 重要通知：本作因重大调整需要，将进入较长的维护期。具体恢复时间另行通知。对于给各位读者带来的不便，深表歉意。
